SECTION 1
What Is The MVP?
Scope definition and what does NOT belong in V1
The MVP (Minimum Viable Product) is the smallest version of Mapping Nexus that delivers real, measurable 
value to a manager at a real company — without needing any cross-company data, any benchmarking layer, 
or any enterprise-grade compliance infrastructure. It is Layer 1 only. Everything else is post-MVP.
Core rule: The MVP is finished when a manager uses it to make a real assignment decision and you 
can document the time or money saved. Not when the code compiles. Not when the demo looks 
good. When a real manager uses it on a real assignment.
What IS in the MVP
• Synthetic data pipeline for building and testing without client data
• Employee profile schema — the data model everything queries against
• AI-powered matching engine using RAG architecture
• Nexus Map — live visual workforce map with utilisation heatmap
• Burnout Radar — early overload signal detection
• Skill Pulse — dormant and in-demand skill tracking
• Role-Based Access Control (RBAC) — three user roles
• Basic privacy architecture — name encryption, DPA template, opt-in telemetry

Technical Architecture Terms
Term Plain Definition
RAG (Retrieval Augmented 
Generation)
The AI architecture that prevents hallucination. Instead of the AI 
generating answers from its own memory (which causes it to invent 
names and scores), RAG first retrieves real employee records from 
your database, then has the AI reason only over those retrieved 
facts. Hallucination drops to near-zero.
LLM (Large Language 
Model)
The AI reasoning layer — e.g. GPT-4, Claude. In this architecture, 
the LLM never generates from memory. It only reasons over records 
retrieved by the RAG pipeline.
Hallucination When an AI invents information that is not in the data — for example,
making up an employee's name or performance score. RAG 
architecture eliminates this by forcing the AI to work only from 
retrieved records.
Vector Database A specialised database that stores employee profiles as 
mathematical representations (vectors) so the system can find 
semantically similar profiles quickly. For example, a search for 
'Python developer with regulatory experience' will find relevant 
employees even if their profiles don't use those exact words.
Embedding The process of converting text (e.g. an employee skill list) into a 
numerical vector that can be compared mathematically. Two 
employees with similar skill sets will have embeddings that are close 
together in vector space.
Docker Container A packaged, portable software unit that includes your AI agent and 
all its dependencies. You ship this container to a client — they run it 
on their own server. You never touch their data. The container is the 
product delivery mechanism for on-prem deployment (post-MVP).
On-Prem (On-Premise) The AI runs inside the client's own server infrastructure, not on your 
cloud. Their employee data never leaves their building. This is the 
enterprise-grade deployment model for Option 1 architecture — 
planned for post-MVP scale.
Telemetry Anonymised performance signals sent back from a client's 
deployment to help you understand model quality. Includes: query 
type, response latency, thumbs up/down, hallucination flag. Never 
includes names, scores, or any employee records.
RBAC (Role-Based Access 
Control)
A security system where each user only sees data relevant to their 
role. Managers see team assignments. HR sees workforce analytics.
Employees see only their own profile. No user can access data 
outside their role.
API (Application 
Programming Interface)
The technical connection layer that allows different software systems
to communicate. Your matching engine exposes an API so the 
frontend dashboard can query it.
Schema The defined structure of your database — what fields an employee 
profile contains, what data type each field accepts, and what is 
mandatory vs optional. Getting this right early is critical because 
changing it later is expensive.
AES-256 Encryption The encryption standard used to protect stored data. AES-256 
means the data is encoded with a 256-bit key — the current gold

FEATURE 3
AI Matching Engine (RAG Architecture)
This is the product. A manager describes a project requirement. The engine retrieves relevant employee 
records from the database, passes them to the LLM, and returns a ranked shortlist with confidence scores 
and explainability — all without the LLM ever generating from memory.
Component What it does
Manager inputs Skills needed, timeline, budget ceiling, seniority level, availability 
window, location constraints, travel requirement
Step 1 — Retrieval The system queries the vector database for the top 10-20 most 
semantically relevant employee profiles based on the project 
requirements
Step 2 — Reasoning The LLM receives the retrieved profiles and the project requirements.
It ranks them, scores them, and generates an explanation for each 
ranking.
Step 3 — Output Ranked shortlist of 3-5 employees, each with: confidence score (%), 
reason for ranking, availability status, cost per day, and ROI estimate
vs. average assignment
Hallucination preventionSystem prompt explicitly instructs: 'Answer only from the provided 
data. If information is not in the retrieved records, say: Data not 
available. Never invent names or scores.'
Confidence score logic Weighted combination of: skill match (40%), availability match (25%),
cost fit (20%), seniority match (15%)
Explainability example Ranked #1: Priya Sharma — Python (Expert, used 3 weeks ago), 
available from Monday, Rs.3,800/day (12% under budget), led 2 
similar analytics projects last year.

FEATURE 5
Burnout Radar
Detects early overload signals before they translate into resignations or performance drops. This feature has 
a direct, quantifiable ROI: preventing one resignation saves approximately one year's salary in rehiring and 
productivity loss — making it one of the easiest features to justify to a CFO.
Signal What it measures
Task Velocity Number of tasks assigned per week. If above the employee's 90-day 
average by more than 30%, flag is raised.
No Leave Taken Employees who have not taken leave in 60+ days despite high task 
load are flagged as high burnout risk.
Consecutive AssignmentsEmployees assigned to 3 or more back-to-back projects with less 
than 5 working days between them.
Overtime Proxy If an employee is logging more tasks per day than their assigned 
capacity suggests is possible, flag for review.

FEATURE 6
Skill Pulse
A weekly view for HR and managers showing which skills across the workforce are going dormant and which 
are in high demand across active projects. This enables proactive internal talent deployment before a hiring 
decision is made.
Element Description
Dormant Skills Panel Skills that have not appeared on any active project assignment in 
90+ days. Listed by employee with last-used date.
In-Demand Skills Panel Skills appearing on the highest number of active and upcoming 
projects. Shows the gap between demand and available supply.
Dormancy threshold Configurable — default 90 days. HR can adjust per role type (e.g., 
clinical trial skills may have natural 6-month dormancy cycles).
Action triggers '3 employees have dormant SAP skills. 2 upcoming projects require 
SAP. Recommend: assign before skills fully atrophy.'
Refresh cadence Weekly batch update. Not real-time — weekly review is the natural 
usage rhythm for this feature.
Primary user HR Personnel — used as a weekly planning ritual. Secondary: 
department managers.

FEATURE 8
Privacy Architecture (MVP-Tier)
The MVP privacy architecture is not a full enterprise compliance stack — that comes after the first paying 
client. But it is enough to walk into a first enterprise meeting without the security objection killing the 
conversation.
Component What you build for the MVP
Name Encryption Employee names stored encrypted using AES-256. The decryption 
key is held by the client organisation, not Mapping Nexus. Even your 
own engineers cannot read employee names.
DPA Template A pre-drafted Data Processing Agreement ready to send at the first 
enterprise meeting. Defines Mapping Nexus as data processor, client
as data controller, and specifies data handling obligations.
Opt-in Telemetry Clients can opt in to sending anonymised usage signals: query type, 
response latency, thumbs up/down, hallucination flag, model version.
Zero PII. Zero employee records. Just quality signals.
DPDP Alignment Data residency in India. Right to deletion within 30 days on request. 
Explicit consent recorded at onboarding. These three steps make 
you DPDP Act 2023 compliant without a formal certification process.
Zero Raw Data Policy Codified in writing: no raw employee data is ever sold or shared by 
any Mapping Nexus team member. Applies to all staff, contractors, 
and third parties. Written into employment contracts from Day 1.
What you do NOT build yetSOC 2 Type II audit (6-12 months), ISO 27001 certification, Business
Associate Agreements for regulated industries, full pen-testing 
infrastructure. These follow after first paying client.

SECTION 5
Recommended Tech Stack
What to use and why — optimised for speed, not perfection
These recommendations are optimised for getting to a demo-ready MVP as fast as possible. They are not 
necessarily what you will use at 10,000 employees per client. You will refactor. That is fine. Ship first.
Layer Recommended Tool & Reason
Backend Language Python — fastest for AI/ML work. Faker library for synthetic data. 
Direct LangChain integration for RAG pipeline.
Database PostgreSQL with pgvector extension — handles both relational 
employee data and vector embeddings in one database. Reduces 
infrastructure complexity at MVP stage.
Vector Search pgvector (built into PostgreSQL) for MVP. Migrate to Pinecone or 
Weaviate at scale if query volume demands it.
LLM API Anthropic Claude API or OpenAI GPT-4 — both support the RAG 
pattern well. Use Anthropic for stronger instruction-following on the 
'never invent data' guardrail.
RAG Framework LangChain — the fastest way to build and iterate on a RAG pipeline. 
Well-documented, large community.
Frontend React + Tailwind CSS — fast to build, large talent pool in India, good
charting library ecosystem.
Visualisation (Nexus Map)D3.js or Recharts — both handle the heatmap and network graph 
requirements. Recharts is faster to prototype; D3 is more flexible for 
the Nexus Map visual.
Authentication & RBAC Supabase Auth or Auth0 — both provide role-based access control 
out of the box. Supabase also handles the PostgreSQL layer, 
reducing stack complexity.
Hosting (MVP) Railway or Render for backend. Vercel for frontend. Both are free tier
until you have paying clients.
Encryption Python's cryptography library (Fernet for AES-256). Client key 
management via a simple key handoff process at onboarding for 
MVP — move to AWS KMS at scale.
Telemetry Simple POST endpoint to your own logging database. No third-party 
analytics tool at MVP stage — keep the data footprint minimal.

SECTION 6
Immediate Actions: First 30 Days
What to do before writing a single line of product code
1. Action 1 — Lock the co-founder equity split, vesting schedule, and IP assignment agreement. This 
conversation becomes impossible to have cleanly once you have users or investors. Do it this week.
2. Action 2 — Define and document the employee data schema. Every engineer you hire, every design 
partner you sign, and every VC you pitch will ask about it. It needs to exist in writing before the first 
line of code.
3. Action 3 — Build the synthetic data generator. Two days of Python work. You now have 1,000 test 
employees and zero data liability. Everything you build from this point is testable.
4. Action 4 — Draft your Design Partner Agreement and DPA template. Get them legally reviewed. 
Have them ready to send before you approach your first target company. Arriving at a meeting 
without these is leaving deals on the table.
5. Action 5 — Identify 30 target design partner companies in your beachhead (100-500 employee 
Indian tech, pharma, or manufacturing firms). Start warm outreach. Target: 10 discovery calls booked 
within 30 days.
6. Action 6 — Source 5 public workforce research datasets (LinkedIn Workforce Report India, 
McKinsey, Mercer, SHRM, Gartner HR Research). Load them into your benchmark baseline layer — 
labelled clearly as Industry Research Baseline. This gives you something to show at early demos 
before you have real data.
The single most important near-term milestone: Three real companies using the product on real 
assignments, with documented hours saved. This is what turns the MVP into a fundable pre-seed 
story. Everything before that is preparation.
